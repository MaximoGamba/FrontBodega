src/
│
├── main.jsx              → Punto de entrada. Monta Provider Redux, BrowserRouter y ToastContainer
├── App.jsx               → Define todas las rutas, lazy loading, guards y ErrorBoundaries por ruta
├── index.css             → Variables CSS globales (colores, fuentes) y keyframes (spin, parpadeo-stock)
│
├── assets/               → Archivos estáticos importados directamente en el código
│   ├── hero.png          → Imagen del hero de la landing
│   ├── react.svg         → Logo React (generado por Vite, sin uso real)
│   └── vite.svg          → Logo Vite (generado por Vite, sin uso real)
│
├── data/                 → Constantes estáticas de contenido. Evitan hardcodear texto en JSX
│   ├── contacto.js       → Array INFO_CONTACTO: teléfono, email, horario del footer
│   ├── historia.js       → Textos, secciones y números de la página Historia
│   ├── links.js          → Array NAV_LINKS: rutas y etiquetas de navegación. Usado por NavLinks y FooterLinks
│   └── pago.js           → DATOS_TRANSFERENCIA: CBU, alias y titular para pagos por transferencia
│
├── styles/               → Objetos de estilo JS reutilizables entre componentes del mismo dominio
│   ├── inputStyles.js    → Base de todos los estilos: authInputStyle, formInputStyle, adminInputStyle, profileInputStyle
│   ├── adminStyles.js    → Re-exporta inputStyle, labelStyle, errorStyle desde inputStyles para el panel admin
│   ├── authStyles.js     → Re-exporta inputStyle, labelStyle desde inputStyles para login y registro
│   └── profileStyles.js  → Re-exporta inputStyle, labelStyle desde inputStyles para el perfil de usuario
│
├── utils/                → Funciones puras sin estado. No saben nada de Redux ni React
│   ├── roles.js          → Constante ROL_ADMIN = "admin"
│   ├── formatters.js     → Máscaras de input (tarjeta, vencimiento), soloNumeros, calcularPrecioFinal, calcularSubtotal
│   ├── validators.js     → validarEmail, validarVencimiento, validarProducto (formulario admin)
│   ├── productosSort.js  → ordenarProductos (por precio/fecha, siempre mueve sin stock al final) y sortProductosAdmin (admin)
│   └── pedidoUtils.js    → Helpers de display para estados de pedido y envío (labels y colores por estado)
│
├── redux/                → Todo el estado global de la app (Redux Toolkit)
│   │
│   ├── store.js          → Configura el store con los 10 slices. Define qué se persiste en localStorage.
│   │                        El objeto storage (adapter inline) va declarado después de todos los imports
│   │                        para respetar las reglas de módulos ES
│   ├── api.js            → Instancia Axios compartida + injectStore para romper dependencia circular.
│   │                        Interceptor de request: agrega JWT y verifica expiración con tokenExpirado().
│   │                        Interceptor de response: desenvuelve .data, maneja 401 con logout automático
│   │
│   │   ── Slices de dominio (datos del servidor) ──
│   ├── authSlice.js      → Sesión: usuario logueado + JWT token. Thunks: login, registro. Persiste en localStorage
│   ├── vinosSlice.js     → Catálogo de vinos. Dos EntityAdapters: público (clientes) y admin.
│   │                        7 thunks CRUD. Selector memoizado selectVinosFiltrados (filtros + búsqueda + orden).
│   │                        TTL con statusAt. loadingMutacion/errorMutacion rastrean pending/rejected de
│   │                        postVino y putVino. putVino.fulfilled remueve del adapter público si el vino está inactivo
│   ├── catalogosSlice.js → 6 arrays de opciones: colores, cepas, azúcares, crianzas, elaboraciones, medidas.
│   │                        Cargados en paralelo con Promise.all. TTL con statusAt
│   ├── pedidosSlice.js   → Pedidos para el panel admin. EntityAdapter. Escucha logout para limpiar datos. TTL
│   ├── usuarioSlice.js   → Datos de perfil + historial de pedidos del usuario logueado.
│   │                        statusDatos ("idle"/"loading"/"succeeded"/"failed") reemplaza al boolean loading.
│   │                        newsletterSuscripto indica si el usuario ya se suscribió al newsletter.
│   │                        Todo se limpia en logout (incluido newsletterSuscripto y statusDatos → "idle")
│   ├── carritoSlice.js   → Items del carrito. Controla stock máximo. Se vacía en logout. Persiste en localStorage
│   ├── checkoutSlice.js  → Wizard de compra: formularios de envío y pago, paso actual, estado del proceso.
│   │                        Thunk procesarCheckout: orquesta crearPedido → crearEnvio → crearPago con rollback
│   ├── galeriaSlice.js   → Estado de upload de imágenes a Cloudinary. Rastrea subiendo: boolean y error: string|null
│   │
│   │   ── Slices de UI (estado de pantalla, sin datos del servidor) ──
│   ├── catalogoUISlice.js → Filtros del catálogo (checkboxes, rango de precio), búsqueda, orden. Constante PRECIO_MAX
│   └── adminUISlice.js    → Tab activo en admin (productos/órdenes), id del producto en edición, visibilidad del form.
│                             ordenesExpandidas (array de IDs de pedidos expandidos). toggleOrdenExpandida para FilaOrden
│
├── services/             → Capa de acceso a la API. Funciones async puras, sin Redux. Una por microservicio
│   ├── authService.js    → loginAPI, registrarAPI → microservicio VITE_AUTH_URL
│   ├── vinosService.js   → fetchVinos, fetchVinosAdmin, fetchVinoById, crearVino, actualizarVino, desactivar/reactivar
│   │                        + función mapVino (normaliza la respuesta del servidor) → microservicio VITE_VINOS_URL
│   ├── catalogosService.js → fetchColores, fetchCepas, fetchAzucares, fetchCrianzas, fetchElaboraciones, fetchMedidas
│   │                         → microservicio VITE_CATALOGOS_URL
│   ├── pedidosService.js → fetchPedidosAdmin, crearPedido, actualizarEstadoPedido → microservicio VITE_PEDIDOS_URL
│   ├── enviosService.js  → crearEnvio (asocia dirección a un pedido) → microservicio VITE_PEDIDOS_URL
│   ├── pagosService.js   → crearPago (asocia monto y método a un pedido) → microservicio VITE_PEDIDOS_URL
│   ├── usuariosService.js → fetchUsuario, actualizarUsuario, fetchPedidosUsuario → microservicio VITE_USUARIOS_URL
│   └── cloudinaryService.js → uploadImagen: sube archivo directo a Cloudinary via fetch (sin JWT, API externa)
│
├── hooks/                → Mediadores entre Redux y los componentes. Encapsulan "cuándo y cómo pedir datos"
│   ├── useVinos.js       → Carga el catálogo público. Patrón TTL: refetch automático si los datos tienen +5 min
│   ├── useVinosAdmin.js  → Igual que useVinos pero para la colección admin (incluye inactivos). TTL 5 min
│   ├── useVino.js        → Carga un vino por ID. Solo fetcha si no está ya en el adapter (cache local)
│   ├── useCatalogos.js   → Carga los 6 catálogos. TTL 5 min. useMemo para estabilizar referencia del objeto
│   │                        y evitar re-renders en cascada en formularios que dependen de las opciones
│   ├── usePedidosAdmin.js → Carga pedidos del panel admin. TTL 5 min
│   ├── usePerfil.js      → Carga datos de perfil y pedidos del usuario. Dos efectos separados:
│   │                        perfil solo si statusDatos === "idle" (evita refetch redundante en cada mount),
│   │                        pedidos solo si statusPedidos === "idle" y no es admin.
│   │                        esAdmin se deriva internamente desde state.auth.usuario?.rol (no recibe parámetro).
│   │                        Retorna cargando: statusDatos === "loading", consistente con todos los demás hooks
│   ├── useProductCard.js → Combina: precio final con descuento, cantidad en carrito, si hay stock disponible,
│   │                        y la función de agregar al carrito. Usado por ProductCard
│   └── useAgregarAlCarrito.js → Verifica si el usuario está logueado antes de agregar. Redirige a /login si no

├── views/                → Páginas completas. Capa delgada: componen componentes y conectan Redux vía hooks

Cómo leer el diagrama:

  redux/     es el corazón del estado. Todo fluye desde ahí.
  services/  es la única capa que habla con el backend. Nada más lo hace directamente.
  hooks/     son el puente: los componentes no tocan Redux directamente para pedir datos.
  views/     son páginas completas pero delgadas: solo componen componentes y usan hooks.
  components/ son las piezas visuales. Reciben props, no saben de dónde vienen los datos.
  utils/ y data/ son utilidades puras: no dependen de React ni Redux.
